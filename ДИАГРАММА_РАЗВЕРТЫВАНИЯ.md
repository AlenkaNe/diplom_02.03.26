# Диаграмма развертывания веб-приложения "Здоровое питание"

## Общее описание архитектуры

Веб-приложение "Здоровое питание" представляет собой трехзвенную архитектуру (3-tier architecture), состоящую из:
1. **Уровень представления (Presentation Layer)** - Frontend сервер
2. **Уровень бизнес-логики (Business Logic Layer)** - Backend API сервер
3. **Уровень данных (Data Layer)** - База данных PostgreSQL

Дополнительно система интегрируется с внешним API для получения данных о продуктах питания.

---

## Компоненты системы

### 1. Клиентское приложение (Web Browser)
**Тип:** Веб-браузер пользователя

**Технологии:**
- HTML5
- CSS3
- JavaScript (Vanilla JS)
- Fetch API для HTTP-запросов

**Функции:**
- Отображение пользовательского интерфейса
- Отправка HTTP-запросов к Frontend серверу
- Обработка ответов и динамическое обновление контента
- Управление состоянием авторизации через cookies

**Протоколы:**
- HTTP/HTTPS
- WebSocket (не используется в текущей версии)

---

### 2. Frontend сервер (Express.js)
**Тип:** Node.js приложение (Express framework)

**Порт:** 3001 (настраивается через переменную окружения PORT)

**Технологии:**
- Node.js
- Express.js 4.18.2
- http-proxy-middleware 2.0.6
- cookie-parser 1.4.6
- jsonwebtoken 9.0.2
- bcryptjs 2.4.3
- pg (PostgreSQL клиент) 8.11.3
- dotenv 17.2.3

**Функции:**
- Раздача статических файлов (HTML, CSS, JavaScript)
- Проксирование запросов к Backend API (`/foods/*` → `http://localhost:3000`)
- Обработка аутентификации и авторизации пользователей
- Управление сессиями через JWT токены (хранятся в HTTP-only cookies)
- API endpoints для работы с пользователями:
  - `POST /api/register` - регистрация нового пользователя
  - `POST /api/login` - вход в систему
  - `POST /api/logout` - выход из системы
  - `GET /api/me` - получение информации о текущем пользователе
  - `PUT /api/profile` - обновление профиля пользователя
  - `PUT /api/me/calculator` - сохранение результатов калькулятора
- API endpoints для работы с дневником питания:
  - `GET /api/meal-types` - получение типов приемов пищи
  - `POST /api/diary/add` - добавление продукта в дневник
  - `GET /api/diary` - получение записей дневника за дату
- Прямое подключение к базе данных PostgreSQL для работы с пользователями и дневником

**Конфигурация:**
- Переменные окружения (`.env`):
  - `PORT` - порт сервера (по умолчанию 3001)
  - `JWT_SECRET` - секретный ключ для подписи JWT токенов
  - `NODE_ENV` - окружение (development/production)

**Развертывание:**
- Запуск: `npm start` или `node server.js`
- Разработка: `npm run dev` (с nodemon для автоперезагрузки)

---

### 3. Backend API сервер (NestJS)
**Тип:** Node.js приложение (NestJS framework)

**Порт:** 3000 (настраивается через переменную окружения PORT)

**Технологии:**
- Node.js
- NestJS 11.0.1
- TypeScript 5.7.3
- @nestjs/axios 4.0.1
- Axios 1.10.0
- @nestjs/config 4.0.2

**Функции:**
- Обработка запросов поиска продуктов питания
- Интеграция с внешним API FoodData Central (USDA)
- Преобразование и форматирование данных о продуктах
- API endpoint: `GET /foods/search` - поиск продуктов по запросу

**Конфигурация:**
- Переменные окружения (`.env`):
  - `PORT` - порт сервера (по умолчанию 3000)
  - `FOOD_DATA_API_KEY` - API ключ для доступа к FoodData Central API

**Развертывание:**
- Разработка: `npm run start:dev` (с автоперезагрузкой)
- Продакшн: `npm run build` → `npm run start:prod`
- Сборка: TypeScript компилируется в JavaScript в папку `dist/`

---

### 4. База данных PostgreSQL
**Тип:** Реляционная база данных

**Порт:** 5432 (стандартный порт PostgreSQL)

**Конфигурация подключения:**
- Host: localhost
- Database: food_project
- User: postgres
- Password: postgres (должно быть изменено в продакшн)

**Схема базы данных:**

**Таблица `users`:**
- `id` (PRIMARY KEY, SERIAL) - уникальный идентификатор пользователя
- `login` (VARCHAR, UNIQUE, NOT NULL) - логин пользователя
- `password_hash` (VARCHAR, NOT NULL) - хеш пароля (bcrypt)
- `first_name` (VARCHAR, NULL) - имя пользователя
- `last_name` (VARCHAR, NULL) - фамилия пользователя
- `gender` (VARCHAR, NULL) - пол (мужчина/женщина)
- `age` (INTEGER, NULL) - возраст
- `height_cm` (INTEGER, NULL) - рост в сантиметрах
- `weight` (DECIMAL, NULL) - вес в килограммах
- `created_at` (TIMESTAMP) - дата создания записи

**Таблица `user_metrics`:**
- `id` (PRIMARY KEY, SERIAL) - уникальный идентификатор
- `user_id` (FOREIGN KEY → users.id) - ссылка на пользователя
- `goal` (VARCHAR, NULL) - цель (похудение/поддержание/набор)
- `activity_level` (VARCHAR, NULL) - уровень активности
- `daily_kcal_norm` (INTEGER, NULL) - суточная норма калорий
- `body_mass_index` (DECIMAL, NULL) - индекс массы тела
- `daily_protein` (DECIMAL, NULL) - норма белков
- `daily_fat` (DECIMAL, NULL) - норма жиров
- `daily_carbs` (DECIMAL, NULL) - норма углеводов

**Таблица `meal_types`:**
- `id` (PRIMARY KEY, SERIAL) - уникальный идентификатор
- `name` (VARCHAR, NOT NULL) - название типа приема пищи (Завтрак, Обед, Ужин, Перекус)

**Таблица `diary_entries`:**
- `id` (PRIMARY KEY, SERIAL) - уникальный идентификатор
- `user_id` (FOREIGN KEY → users.id) - ссылка на пользователя
- `meal_type_id` (FOREIGN KEY → meal_types.id) - тип приема пищи
- `food_id` (INTEGER, NULL) - ID продукта из внешнего API (опционально)
- `amount_gram` (DECIMAL, NOT NULL) - вес продукта в граммах
- `custom_name` (VARCHAR, NOT NULL) - название продукта
- `custom_kcal` (DECIMAL, NULL) - калории на 100г
- `custom_protein` (DECIMAL, NULL) - белки на 100г
- `custom_fat` (DECIMAL, NULL) - жиры на 100г
- `custom_carbs` (DECIMAL, NULL) - углеводы на 100г
- `eaten_at` (DATE, NOT NULL) - дата приема пищи
- `added_at` (TIMESTAMP) - дата и время добавления записи

**Подключение:**
- Frontend сервер подключается к базе данных через пул соединений (pg.Pool)
- Автоматическая проверка подключения при запуске приложения

---

### 5. Внешний API - FoodData Central (USDA)
**Тип:** Внешний REST API

**URL:** https://api.nal.usda.gov/fdc/v1

**Протокол:** HTTPS

**Аутентификация:** API Key (передается как query параметр `api_key`)

**Функции:**
- Предоставление данных о продуктах питания
- Поиск продуктов по названию
- Получение детальной информации о питательной ценности продуктов

**Endpoint:**
- `GET /foods/search` - поиск продуктов с параметрами:
  - `query` - поисковый запрос
  - `pageSize` - количество результатов на странице
  - `pageNumber` - номер страницы
  - `api_key` - API ключ

**Интеграция:**
- Backend сервер (NestJS) отправляет HTTP-запросы к этому API
- Данные преобразуются в формат приложения через FoodService

## Потоки данных

### 1. Поиск продуктов питания

```
Клиент → Frontend (3001) → Backend (3000) → FoodData Central API
                                                      ↓
Клиент ← Frontend (3001) ← Backend (3000) ← Данные о продуктах
```

**Описание:**
1. Пользователь вводит запрос в форму поиска
2. JavaScript отправляет запрос на `POST /foods/search` (через Frontend сервер)
3. Frontend сервер проксирует запрос на Backend сервер (`http://localhost:3000/foods/search`)
4. Backend сервер отправляет запрос к FoodData Central API с API ключом
5. Внешний API возвращает результаты поиска
6. Backend сервер преобразует данные в формат приложения
7. Данные возвращаются через Frontend сервер клиенту
8. JavaScript отображает результаты в таблице

---

### 2. Регистрация пользователя

```
Клиент → Frontend (3001) → PostgreSQL
              ↓
         JWT токен в cookie
              ↓
         Клиент (авторизован)
```

**Описание:**
1. Пользователь заполняет форму регистрации
2. Данные отправляются на `POST /api/register`
3. Frontend сервер хеширует пароль (bcrypt)
4. Данные сохраняются в таблицу `users` в PostgreSQL
5. Создается JWT токен с `userId`
6. Токен сохраняется в HTTP-only cookie
7. Пользователь автоматически авторизован

---

### 3. Вход в систему

```
Клиент → Frontend (3001) → PostgreSQL (проверка пароля)
              ↓
         JWT токен в cookie
              ↓
         Клиент (авторизован)
```

**Описание:**
1. Пользователь вводит логин и пароль
2. Данные отправляются на `POST /api/login`
3. Frontend сервер ищет пользователя в базе данных по логину
4. Сравнивается хеш пароля из базы с введенным паролем (bcrypt.compare)
5. При успехе создается JWT токен
6. Токен сохраняется в HTTP-only cookie
7. Пользователь авторизован

---

### 4. Добавление продукта в дневник

```
Клиент → Frontend (3001) → PostgreSQL (diary_entries)
              ↓
         Успешное добавление
              ↓
         Обновление дневника
```

**Описание:**
1. Пользователь выбирает продукт и нажимает "Добавить в дневник"
2. Открывается модальное окно с формой
3. Пользователь выбирает тип приема пищи и указывает вес
4. Данные отправляются на `POST /api/diary/add` (с JWT токеном в cookie)
5. Frontend сервер проверяет JWT токен (authMiddleware)
6. Данные сохраняются в таблицу `diary_entries`
7. Возвращается подтверждение успешного добавления

---

### 5. Просмотр дневника питания

```
Клиент → Frontend (3001) → PostgreSQL (diary_entries, meal_types)
              ↓
         Записи за выбранную дату
              ↓
         Группировка по типам приемов пищи
              ↓
         Отображение с итогами
```

**Описание:**
1. Пользователь выбирает дату в календаре
2. JavaScript отправляет запрос на `GET /api/diary?date=YYYY-MM-DD`
3. Frontend сервер проверяет JWT токен
4. Выполняется SQL-запрос к базе данных:
   - JOIN между `diary_entries` и `meal_types`
   - Фильтрация по `user_id` и `eaten_at`
5. Данные группируются по типам приемов пищи
6. Рассчитываются итоги за день (калории, белки, жиры, углеводы)
7. Данные возвращаются клиенту и отображаются в секциях

---

### 6. Расчет калорий (калькулятор)

```
Клиент (ввод данных) → Frontend (3001) → PostgreSQL (user_metrics)
              ↓
         Расчет на клиенте
              ↓
         Сохранение результатов
```

**Описание:**
1. Пользователь заполняет форму калькулятора (пол, возраст, рост, вес, активность, цель, формула)
2. JavaScript выполняет расчет на стороне клиента:
   - Расчет базового метаболизма (BMR) по формуле Харриса-Бенедикта или Миффлина-Сан Жеора
   - Расчет общего расхода энергии (TDEE) = BMR × коэффициент активности
   - Корректировка в зависимости от цели (похудение/поддержание/набор)
   - Расчет макронутриентов (белки, жиры, углеводы)
   - Расчет индекса массы тела (ИМТ)
3. Результаты отображаются на странице
4. Если пользователь авторизован, данные сохраняются через `PUT /api/me/calculator`
5. Данные сохраняются в таблицу `user_metrics` (или обновляются, если уже существуют)

## Требования к развертыванию

### Системные требования

**Frontend сервер:**
- Node.js версии 14.0.0 или выше
- npm версии 6.0.0 или выше
- Минимум 512 MB RAM
- Минимум 100 MB свободного места на диске

**Backend сервер:**
- Node.js версии 14.0.0 или выше
- npm версии 6.0.0 или выше
- TypeScript компилятор (устанавливается через npm)
- Минимум 512 MB RAM
- Минимум 200 MB свободного места на диске (включая `dist/`)

**База данных:**
- PostgreSQL версии 12.0 или выше
- Минимум 1 GB RAM
- Минимум 500 MB свободного места на диске
- Созданная база данных `food_project`
- Настроенные таблицы (users, user_metrics, meal_types, diary_entries)

### Порядок развертывания

1. **Установка зависимостей:**
   ```bash
   # Frontend
   cd frontend
   npm install
   
   # Backend
   cd ../backend
   npm install
   ```

2. **Настройка базы данных:**
   - Установить и запустить PostgreSQL
   - Создать базу данных `food_project`
   - Выполнить SQL-скрипты для создания таблиц
   - Настроить подключение в `frontend/db.js`

3. **Настройка переменных окружения:**
   - Создать `.env` файлы в `frontend/` и `backend/`
   - Заполнить необходимые переменные

4. **Сборка Backend:**
   ```bash
   cd backend
   npm run build
   ```

5. **Запуск серверов:**
   ```bash
   # Backend (в отдельном терминале)
   cd backend
   npm run start:prod
   
   # Frontend (в отдельном терминале)
   cd frontend
   npm start
   ```

6. **Проверка:**
   - Открыть браузер: `http://localhost:3001`
   - Проверить доступность Backend API: `http://localhost:3000/foods/search?query=apple&api_key=...`

## Безопасность

### Меры безопасности

1. **Аутентификация:**
   - Пароли хранятся в виде bcrypt хешей (12 раундов)
   - JWT токены хранятся в HTTP-only cookies (недоступны через JavaScript)
   - Токены имеют срок действия (30 дней)

2. **Защита от SQL-инъекций:**
   - Использование параметризованных запросов (prepared statements) через pg.Pool
   - Все пользовательские данные экранируются автоматически

3. **CORS:**
   - В текущей версии CORS не настроен (приложение работает на localhost)
   - Для продакшн необходимо настроить CORS политику

4. **API ключи:**
   - API ключ для FoodData Central хранится в переменных окружения
   - Не должен попадать в систему контроля версий (git)

5. **HTTPS:**
   - В продакшн необходимо использовать HTTPS
   - Настроить SSL/TLS сертификаты

## Масштабируемость

### Текущая архитектура (односерверная)

- Все компоненты могут работать на одной машине
- Подходит для небольших нагрузок (до 100 одновременных пользователей)

### Возможности масштабирования

1. **Горизонтальное масштабирование Frontend:**
   - Запуск нескольких экземпляров Frontend сервера
   - Использование балансировщика нагрузки (nginx, HAProxy)
   - Общий Redis для хранения сессий (если потребуется)

2. **Горизонтальное масштабирование Backend:**
   - Запуск нескольких экземпляров Backend сервера
   - Балансировка нагрузки

3. **Масштабирование базы данных:**
   - Репликация PostgreSQL (master-slave)
   - Использование connection pooling (уже реализовано через pg.Pool)
   - Кэширование часто запрашиваемых данных (Redis)

4. **Разделение на микросервисы:**
   - Выделение сервиса аутентификации
   - Выделение сервиса дневника питания
   - Выделение сервиса поиска продуктов

## Заключение

Диаграмма развертывания демонстрирует классическую трехзвенную архитектуру веб-приложения с разделением на Frontend и Backend серверы. Система использует современные технологии (Node.js, Express, NestJS, PostgreSQL) и интегрируется с внешним API для получения данных о продуктах питания.

Архитектура обеспечивает:
