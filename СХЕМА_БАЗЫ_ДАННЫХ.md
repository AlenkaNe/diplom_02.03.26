## Описание таблиц

### 1. Таблица `users`

**Назначение:** Хранение информации о пользователях системы

**Структура:**

| Колонка | Тип данных | Ограничения | Описание |
|---------|-----------|-------------|----------|
| `id` | SERIAL | PRIMARY KEY, NOT NULL | Уникальный идентификатор пользователя (автоинкремент) |
| `login` | VARCHAR | UNIQUE, NOT NULL | Логин пользователя (уникальный) |
| `password_hash` | VARCHAR | NOT NULL | Хеш пароля пользователя (bcrypt, 12 раундов) |
| `first_name` | VARCHAR | NULL | Имя пользователя |
| `last_name` | VARCHAR | NULL | Фамилия пользователя |
| `gender` | VARCHAR | NULL | Пол пользователя. Допустимые значения: 'мужчина', 'женщина' |
| `age` | INTEGER | NULL | Возраст пользователя (1-120) |
| `height_cm` | INTEGER | NULL | Рост пользователя в сантиметрах (50-250) |
| `weight` | DECIMAL(5,2) | NULL | Вес пользователя в килограммах (20-500) |
| `created_at` | TIMESTAMP | DEFAULT NOW() | Дата и время создания записи |

**Индексы:**
- PRIMARY KEY на `id`
- UNIQUE INDEX на `login`

**Ограничения:**
- `login` должен быть уникальным (код ошибки PostgreSQL: 23505)

**Примеры использования:**
- Регистрация нового пользователя
- Авторизация пользователя (проверка логина и пароля)
- Обновление профиля пользователя
- Получение информации о пользователе

**SQL для создания таблицы:**
```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    login VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    first_name VARCHAR(255),
    last_name VARCHAR(255),
    gender VARCHAR(50),
    age INTEGER CHECK (age >= 1 AND age <= 120),
    height_cm INTEGER CHECK (height_cm >= 50 AND height_cm <= 250),
    weight DECIMAL(5,2) CHECK (weight >= 20 AND weight <= 500),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE UNIQUE INDEX idx_users_login ON users(login);
```

### 2. Таблица `user_metrics`

**Назначение:** Хранение метрик и результатов расчетов пользователя (калькулятор калорий)

**Структура:**

| Колонка | Тип данных | Ограничения | Описание |
|---------|-----------|-------------|----------|
| `id` | SERIAL | PRIMARY KEY, NOT NULL | Уникальный идентификатор записи |
| `user_id` | INTEGER | FOREIGN KEY → users.id, UNIQUE, NOT NULL | Ссылка на пользователя (один к одному) |
| `goal` | VARCHAR | NULL | Цель пользователя. Допустимые значения: 'похудение', 'поддержание', 'набор' |
| `activity_level` | VARCHAR | NULL | Уровень активности. Допустимые значения: 'минимальная активность', 'низкая активность', 'средняя активность', 'высокая активность', 'очень высокая активность' |
| `daily_kcal_norm` | INTEGER | NULL | Суточная норма калорий в килокалориях |
| `body_mass_index` | DECIMAL(4,1) | NULL | Индекс массы тела (ИМТ), округленный до 1 знака после запятой |
| `daily_protein` | DECIMAL(6,2) | NULL | Суточная норма белков в граммах |
| `daily_fat` | DECIMAL(6,2) | NULL | Суточная норма жиров в граммах |
| `daily_carbs` | DECIMAL(6,2) | NULL | Суточная норма углеводов в граммах |

**Связи:**
- FOREIGN KEY `user_id` REFERENCES `users(id)` ON DELETE CASCADE
- UNIQUE constraint на `user_id` (один пользователь - одна запись метрик)

**Индексы:**
- PRIMARY KEY на `id`
- UNIQUE INDEX на `user_id`
- FOREIGN KEY INDEX на `user_id`

**Особенности:**
- Создается автоматически при первом использовании калькулятора
- Обновляется при каждом новом расчете
- Связь один-к-одному с таблицей `users`

**SQL для создания таблицы:**
```sql
CREATE TABLE user_metrics (
    id SERIAL PRIMARY KEY,
    user_id INTEGER UNIQUE NOT NULL,
    goal VARCHAR(50),
    activity_level VARCHAR(100),
    daily_kcal_norm INTEGER,
    body_mass_index DECIMAL(4,1),
    daily_protein DECIMAL(6,2),
    daily_fat DECIMAL(6,2),
    daily_carbs DECIMAL(6,2),
    CONSTRAINT fk_user_metrics_user 
        FOREIGN KEY (user_id) 
        REFERENCES users(id) 
        ON DELETE CASCADE
);

CREATE UNIQUE INDEX idx_user_metrics_user_id ON user_metrics(user_id);
```

**Назначение:** Справочник типов приемов пищи

**Структура:**

| Колонка | Тип данных | Ограничения | Описание |
|---------|-----------|-------------|----------|
| `id` | SERIAL | PRIMARY KEY, NOT NULL | Уникальный идентификатор типа приема пищи |
| `name` | VARCHAR | NOT NULL | Название типа приема пищи |

**Типичные значения:**
- Завтрак
- Обед
- Ужин
- Перекус
- (могут быть добавлены другие типы)

**Индексы:**
- PRIMARY KEY на `id`

**Особенности:**
- Справочная таблица (reference table)
- Данные обычно заполняются при инициализации базы данных
- Используется для группировки записей в дневнике питания

**SQL для создания таблицы:**
```sql
CREATE TABLE meal_types (
    id SERIAL PRIMARY KEY,

-- Заполнение начальными данными
INSERT INTO meal_types (name) VALUES

    ('Ужин'),
    ('Перекус');
```

### 4. Таблица `diary_entries`

**Назначение:** Хранение записей дневника питания пользователей

**Структура:**

| Колонка | Тип данных | Ограничения | Описание |
|---------|-----------|-------------|----------|
| `id` | SERIAL | PRIMARY KEY, NOT NULL | Уникальный идентификатор записи |
| `user_id` | INTEGER | FOREIGN KEY → users.id, NOT NULL | Ссылка на пользователя |
| `meal_type_id` | INTEGER | FOREIGN KEY → meal_types.id, NOT NULL | Тип приема пищи |
| `food_id` | INTEGER | NULL | ID продукта из внешнего API (FoodData Central). Опционально, так как продукты могут добавляться вручную |
| `amount_gram` | DECIMAL(8,2) | NOT NULL | Вес продукта в граммах |
| `custom_name` | VARCHAR(255) | NOT NULL | Название продукта (может быть из API или введено вручную) |
| `custom_kcal` | DECIMAL(8,2) | NULL | Калории на 100 грамм продукта |
| `custom_protein` | DECIMAL(8,2) | NULL | Белки на 100 грамм продукта (в граммах) |
| `custom_fat` | DECIMAL(8,2) | NULL | Жиры на 100 грамм продукта (в граммах) |
| `custom_carbs` | DECIMAL(8,2) | NULL | Углеводы на 100 грамм продукта (в граммах) |
| `eaten_at` | DATE | NOT NULL | Дата приема пищи |
| `added_at` | TIMESTAMP | DEFAULT NOW() | Дата и время добавления записи в дневник |

**Связи:**
- FOREIGN KEY `user_id` REFERENCES `users(id)` ON DELETE CASCADE
- FOREIGN KEY `meal_type_id` REFERENCES `meal_types(id)`
- FOREIGN KEY `food_id` REFERENCES `foods(id)` ON DELETE SET NULL (опционально)

**Индексы:**
- PRIMARY KEY на `id`
- FOREIGN KEY INDEX на `user_id`
- FOREIGN KEY INDEX на `meal_type_id`
- INDEX на `eaten_at` (для быстрого поиска по дате)
- COMPOSITE INDEX на `(user_id, eaten_at)` (для оптимизации запросов дневника)

**Особенности:**
- Поля `custom_*` используются для хранения питательной ценности, так как продукты могут добавляться вручную или из внешнего API
- Расчет нутриентов для конкретной порции выполняется на уровне приложения: `(custom_kcal * amount_gram / 100)`
- Записи группируются по `meal_type_id` и `eaten_at` для отображения в дневнике

**SQL для создания таблицы:**
```sql
CREATE TABLE diary_entries (
    id SERIAL PRIMARY KEY,
    user_id INTEGER NOT NULL,
    meal_type_id INTEGER NOT NULL,
    food_id INTEGER,
    amount_gram DECIMAL(8,2) NOT NULL CHECK (amount_gram > 0),
    custom_name VARCHAR(255) NOT NULL,
    custom_kcal DECIMAL(8,2),
    custom_protein DECIMAL(8,2),
    custom_fat DECIMAL(8,2),
    custom_carbs DECIMAL(8,2),
    eaten_at DATE NOT NULL,
    added_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_diary_entries_user 
        FOREIGN KEY (user_id) 
        REFERENCES users(id) 
        ON DELETE CASCADE,
    CONSTRAINT fk_diary_entries_meal_type 
        FOREIGN KEY (meal_type_id) 
        REFERENCES meal_types(id),
    CONSTRAINT fk_diary_entries_food 
        FOREIGN KEY (food_id) 
        REFERENCES foods(id) 
        ON DELETE SET NULL
);

CREATE INDEX idx_diary_entries_user_id ON diary_entries(user_id);
CREATE INDEX idx_diary_entries_meal_type_id ON diary_entries(meal_type_id);
CREATE INDEX idx_diary_entries_eaten_at ON diary_entries(eaten_at);
CREATE INDEX idx_diary_entries_user_date ON diary_entries(user_id, eaten_at);
```

### 5. Таблица `foods` (опциональная, для будущего использования)

**Назначение:** Кэширование данных о продуктах из внешнего API

**Структура:**

| Колонка | Тип данных | Ограничения | Описание |
|---------|-----------|-------------|----------|
| `id` | INTEGER | PRIMARY KEY, NOT NULL | ID продукта из FoodData Central API (fdcId) |
| `name` | VARCHAR(255) | NOT NULL | Название продукта |
| `kcal_per_100` | DECIMAL(8,2) | NULL | Калории на 100 грамм |
| `protein_g` | DECIMAL(8,2) | NULL | Белки на 100 грамм (в граммах) |
| `fat_g` | DECIMAL(8,2) | NULL | Жиры на 100 грамм (в граммах) |
| `carbs_g` | DECIMAL(8,2) | NULL | Углеводы на 100 грамм (в граммах) |

**Особенности:**
- В текущей версии приложения эта таблица упоминается в запросах, но не используется активно
- Может использоваться для кэширования данных о продуктах, чтобы уменьшить количество запросов к внешнему API
- Связь с `diary_entries` через `food_id` опциональна

**SQL для создания таблицы:**
```sql
CREATE TABLE foods (
    id INTEGER PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    kcal_per_100 DECIMAL(8,2),
    protein_g DECIMAL(8,2),
    fat_g DECIMAL(8,2),
    carbs_g DECIMAL(8,2)
);

CREATE INDEX idx_foods_name ON foods(name);
```

## Типы связей между таблицами

### 1. Связь один-к-одному (1:1)
- **users** ↔ **user_metrics**
  - Один пользователь имеет одну запись метрик
  - Связь через `user_id` с UNIQUE constraint

### 2. Связь один-ко-многим (1:N)
- **users** → **diary_entries**
  - Один пользователь может иметь множество записей в дневнике
  - Связь через `user_id`

- **meal_types** → **diary_entries**
  - Один тип приема пищи может использоваться в множестве записей
  - Связь через `meal_type_id`

- **foods** → **diary_entries** (опционально)
  - Один продукт может использоваться в множестве записей
  - Связь через `food_id` (может быть NULL)

## Основные запросы к базе данных

### 1. Регистрация пользователя
```sql
INSERT INTO users (login, password_hash, first_name, last_name, gender, age, height_cm, weight)
VALUES ($1, $2, $3, $4, $5, $6, $7, $8)
RETURNING id, login, first_name;
```

### 2. Авторизация пользователя
```sql
SELECT id, password_hash 
FROM users 
WHERE login = $1;
```

### 3. Получение информации о пользователе с метриками
```sql
SELECT 
    u.id, u.login, u.first_name, u.last_name, u.gender, u.age, u.height_cm, u.weight,
    um.goal, um.activity_level, um.daily_kcal_norm, um.body_mass_index,
    um.daily_protein, um.daily_fat, um.daily_carbs
FROM users u
LEFT JOIN user_metrics um ON um.user_id = u.id
WHERE u.id = $1;
```

### 4. Обновление/создание метрик пользователя
```sql
-- Обновление (если запись существует)
UPDATE user_metrics
SET goal = $1, activity_level = $2, daily_kcal_norm = $3, ...
WHERE user_id = $4
RETURNING *;

-- Вставка (если записи нет)
INSERT INTO user_metrics (user_id, goal, activity_level, ...)
VALUES ($1, $2, $3, ...)
RETURNING *;
```

### 5. Получение записей дневника за дату
```sql
SELECT 
    de.id,
    de.amount_gram,
    de.eaten_at,
    de.added_at,
    de.custom_name,
    de.custom_kcal,
    de.custom_protein,
    de.custom_fat,
    de.custom_carbs,
    mt.name as meal_type_name,
    f.name as food_name,
    f.kcal_per_100,
    f.protein_g,
    f.fat_g,
    f.carbs_g
FROM diary_entries de
JOIN meal_types mt ON de.meal_type_id = mt.id
LEFT JOIN foods f ON de.food_id = f.id
WHERE de.user_id = $1 AND de.eaten_at = $2
ORDER BY de.eaten_at DESC, mt.id ASC;
```

### 6. Добавление записи в дневник
```sql
INSERT INTO diary_entries 
    (user_id, meal_type_id, amount_gram, custom_name, custom_kcal, 
     custom_protein, custom_fat, custom_carbs, eaten_at)
VALUES ($1, $2, $3, $4, $5, $6, $7, $8, $9)
RETURNING id;
```

## Ограничения целостности данных

### 1. Каскадное удаление
- При удалении пользователя (`users`) автоматически удаляются:
  - Все записи в `user_metrics` (ON DELETE CASCADE)
  - Все записи в `diary_entries` (ON DELETE CASCADE)

### 2. Уникальность
- `users.login` - уникальный логин
- `user_metrics.user_id` - один пользователь - одна запись метрик

### 3. Проверочные ограничения (CHECK)
- `users.age` - от 1 до 120
- `users.height_cm` - от 50 до 250
- `users.weight` - от 20 до 500
- `diary_entries.amount_gram` - больше 0

### 4. Обязательные поля (NOT NULL)
- `users.login`, `users.password_hash`
- `diary_entries.user_id`, `diary_entries.meal_type_id`, `diary_entries.amount_gram`, `diary_entries.custom_name`, `diary_entries.eaten_at`

- `diary_entries.eaten_at` - для фильтрации по дате

   - Кэширование типов приемов пищи (редко изменяются)

4. **Мониторинг:**
   - Отслеживание медленных запросов
   - Анализ использования индексов (EXPLAIN ANALYZE)

## Статистика и объемы данных

### Оценка размеров таблиц

**users:**
- Средний размер записи: ~200 байт
- При 10,000 пользователей: ~2 MB

**user_metrics:**
- Средний размер записи: ~150 байт
- При 10,000 пользователей: ~1.5 MB

**meal_types:**
- Средний размер записи: ~50 байт
- При 4 типах: ~200 байт

**diary_entries:**
- Средний размер записи: ~300 байт
- При 100 записях на пользователя в год и 10,000 пользователей: ~300 MB

**Общий объем (оценка):**
- Для 10,000 активных пользователей: ~300-500 MB
- С учетом индексов: ~500-700 MB
